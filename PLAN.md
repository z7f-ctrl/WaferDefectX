## 1. 生产可用的定义（完成标准）

达到「可生产使用」时，系统至少满足：

| 维度 | 验收标准 |
|------|----------|
| 功能闭环 | 单图输入 → 定位 → 分类 → 结构化报告（含 bbox、类别、置信度、耗时） |
| 运行形态 | C++（或容器服务）可端到端推理；不依赖 notebook / 交互式脚本 |
| 数据可信 | 在真实或逼近真实的数据上报告定位召回、每类漏检/误报；合成集仅作冒烟 |
| 契约稳定 | 路径、模型输入维、类别表、特征版本可复现、可校验 |
| 质量门禁 | CI 冒烟 + 关键单元测试；关键延迟有 P50/P95 基线 |
| 运维可观测 | 日志分级、失败可诊断、配置外置（非硬编码 cwd） |
| 部署可交付 | 明确 Runtime 包（CV / ML / CNN）与依赖清单 |

非目标（本阶段不做完整 fab 系统）：MES 对接、设备协议、多机调度、全自动重训平台。

---

## 2. 优先级总览

```text
P0  工程地基：可安装、可运行、可回归、契约不踩坑
P1  生产闭环：端到端推理路径 + 真实数据评估协议
P2  算法对齐：ROI 语义统一、定位更稳、评估体系 AOI 化
P3  部署增强：服务化、容器、加速与工位开关
```

## 3. P0 — 工程地基（阻塞生产）

目标：任意开发者 / CI 机器能稳定复现训练、导出、推理冒烟；消除已知契约陷阱。

### 3.1 路径与工程结构

- [x] **P0-01** 统一路径解析：所有脚本改为基于 `__file__` / 项目根目录，去掉对 cwd=`code/` + `WaferDefectX/` 前缀的依赖
- [x] **P0-02** 增加 `requirements.txt`（及可选 `requirements-ml.txt`：torch / openvino / hummingbird）
- [x] **P0-03** 理顺 C++ 构建：头文件声明 + `.cpp` 单一编译单元；禁止 `main.cpp` `#include "*.cpp"`
- [x] **P0-04** 提供一键冒烟脚本（如 `scripts/smoke.sh`）：生成/读样张 → Python demo → 导出校验 → C++ CLI

### 3.2 模型与类别契约

- [x] **P0-05** OpenVINO / ONNX 路径：`classes_` 从训练产物或旁路元数据加载，禁止写死 `['good','particle','scratch']`
- [x] **P0-06** 导出时固化契约：输入名/shape、特征维、类别顺序、特征版本号写入 `*.json` 或模型 metadata
- [x] **P0-07** 分类器工厂对缺模型、错 shape、未知 `model_type` 给出明确错误码/信息

### 3.3 质量门禁

- [x] **P0-08** 为 Preprocessor / Localizer / FeatureExtractor 增加最少单元测试（空图、单通道、固定样张特征维=7）
- [x] **P0-09** CI：安装依赖 → 冒烟 → CMake 构建 C++（macOS/Linux 矩阵可后续扩展）
- [x] **P0-10** 文档：README 安装与「从仓库根目录运行」说明与 DESIGN §2.3 对齐

**P0 退出条件**：新 clone 按 README 可完成冒烟；C++ 干净链接；类别表与导出模型一致；路径错误不再成为默认失败模式。 ✅（2026-07-19）

## 4. P1 — 生产闭环（可用旁路工具）

目标：闭合「定位 → 特征 → 推理」生产路径；建立真实数据上的评估协议。

### 4.1 端到端运行时

- [x] **P1-01** 明确部署拓扑（二选一写入 DESIGN ADR，并实现主路径）  
  - **A（进程内）**：C++ 预处理 + 定位 + 特征 + ONNX Runtime / OpenVINO  
  - **B（拆分）**：C++ 定位 → 结构化 ROI → 独立推理服务  
- [x] **P1-02** 实现选定拓扑的端到端 CLI：输入图像路径 → JSON/overlay 输出
- [x] **P1-03** 特征提取 C++ 实现与 Python 数值对齐测试（同一图容差断言）
- [x] **P1-04** 输入校验：分辨率、通道、空图、晶圆 ROI 边界；失败可返回错误结构而非崩溃

### 4.2 训练与标签语义（分类侧）

- [x] **P1-05** 每轮廓独立样本训练/推理（取消「仅最大轮廓」作为默认生产逻辑）
- [x] **P1-06** `good` 上的误检轮廓纳入负类（如 `noise` / `background`），支持拒识
- [x] **P1-07** 训练/评估脚本输出：accuracy + 每类 precision/recall + confusion matrix 落盘

### 4.3 真实数据与评估协议

- [x] **P1-08** 接入真实集解析（优先 WM-811K 或内部样张格式），与合成集目录约定分离
- [x] **P1-09** 定义评估协议文档：划分方式、定位 IoU/命中规则、漏检与误报定义
- [x] **P1-10** 在真实（或 hold-out）集上建立基线报告，合成集 accuracy 不再作为唯一 KPI

**P1 退出条件**：选定拓扑可单命令跑通端到端；有真实/逼近真实数据上的基线指标；负类与多缺陷语义正确。 ✅（2026-07-19）

**P1 退出条件**：选定拓扑可单命令跑通端到端；有真实/逼近真实数据上的基线指标；负类与多缺陷语义正确。

## 5. P2 — 算法与评估对齐（可靠检测）

目标：两条分类路径语义一致；定位更抗产线噪声；指标符合 AOI 决策需求。

### 5.1 检测 / 分类语义统一

- [x] **P2-01** CNN 改为 ROI patch（按 bbox crop → resize 64×64），与经典路径对齐
- [ ] **P2-02** 统一 `DefectClassifier` 输入约定：特征路径 vs patch 路径在 API 层显式区分
- [x] **P2-03** 一体化评估：定位召回、每类漏检、误报率；可选轻量 mAP / F1@IoU

### 5.2 定位管线增强

- [ ] **P2-04** 晶圆圆盘 mask，抑制盘缘 / 视野外伪轮廓
- [ ] **P2-05** 面积、长宽比、圆度等几何先验过滤可配置
- [ ] **P2-06** 自适应阈值路径可开关接入主定位（与 Canny 路径 A/B 可配置）
- [ ] **P2-07** 合成数据增强：暗缺陷、簇状、光照不均、边缘伪影，缩小与真实域差距

### 5.3 性能与可观测

- [ ] **P2-08** 基准升级：单图 P50/P95；分阶段耗时（preprocess / localize / feature / infer）
- [ ] **P2-09** 结构化日志（JSON lines 或统一 `[stage] key=value`），便于产线旁路采集
- [ ] **P2-10** 特征 schema 版本化：改 7 维特征必须 bump 版本并强制重训/重导

**P2 退出条件**：ROI 语义统一；定位在真实样张上误报可控；有 AOI 向指标看板/报告模板。

## 6. P3 — 部署与加速（工位化）

目标：便于部署到工位或边缘盒；按噪声工况开关增强模块。

- [ ] **P3-01** 容器化：Runtime-CV / Runtime-ML / Runtime-CNN 镜像与版本标签
- [ ] **P3-02** 简易 REST 或 gRPC 推理服务（健康检查、同步推理、批量可选）
- [ ] **P3-03** FFT 降噪作为预处理可配置开关（默认关；高噪声工位开启）
- [ ] **P3-04** 多线程批处理 / OpenVINO 吞吐调优；可选 CUDA 路径评估
- [ ] **P3-05** 配置外置：YAML/JSON（阈值、min_area、模型路径、设备）
- [ ] **P3-06** 发布清单：依赖、模型 checksum、兼容 OpenCV/ORT 版本、回滚说明

**P3 退出条件**：可用容器或服务在目标硬件上稳定跑通；配置与模型可版本化发布。
